# Pipeline Contracts

This document defines the **data contracts used across the LLM
pipeline**.

The project follows a **contract-first design**, meaning that every
component of the system exchanges structured data with explicitly
defined schemas.

Contracts serve several purposes:

-   enforce structured communication between components
-   enable automated validation
-   support deterministic evaluation
-   allow reliability failures to be detected automatically

Any violation of these contracts is recorded as a **reliability
failure** during evaluation.

------------------------------------------------------------------------

## Contract Layers

The pipeline defines three main contract layers:

| Contract | Purpose |
|---|---|
| Benchmark Case | Defines the input task evaluated by the system |
| Tool Call | Defines how the model interacts with external tools |
| Pipeline Output | Defines the structured result produced by the system |

------------------------------------------------------------------------

## Benchmark Case Contract

Benchmark cases define the tasks executed by the evaluation framework.

Each case represents a **single evaluation unit**.

### Schema

``` json
{
  "case_id": "string",
  "question": "string",
  "expected_answer": "string"
}
```

### Field Description

| Field | Description |
|---|---|
| `case_id` | Unique identifier for the evaluation case |
| `question` | User query given to the pipeline |
| `expected_answer` | Reference answer used for evaluation |

### Purpose

Benchmark cases allow:

-   deterministic evaluation
-   reproducible experiments
-   cross-model comparisons

They are typically stored as `.jsonl` datasets and loaded by the
evaluation runner.

------------------------------------------------------------------------

## Tool Call Contract

The pipeline allows the model to interact with external tools.

Tool calls must follow a strict structure so that the system can:

-   parse tool requests
-   validate arguments
-   track tool execution

### Schema

``` json
{
  "tool_name": "string",
  "arguments": "object"
}
```

Tool calls are validated before execution to ensure that:

- the tool exists in the registry
- arguments match the expected schema
- the call can be executed safely

### Field Description

| Field | Description |
|---|---|
| `tool_name` | Identifier of the tool to execute |
| `arguments` | JSON object containing tool parameters |

### Example

``` json
{
  "tool_name": "search",
  "arguments": {
    "query": "capital of France"
  }
}
```

### Reliability Implications

Tool calls introduce several possible failure modes:

-   invalid tool name
-   malformed arguments
-   tool execution failure
-   timeout or missing response

These failures are captured by the reliability evaluation framework.

------------------------------------------------------------------------

## Pipeline Output Contract

The final output of the pipeline must follow a structured format.

This allows the system to validate responses and compute reliability
metrics.

### Schema

``` json
{
  "answer": "string",
  "citations": ["string"],
  "tool_calls": []
}
```

### Field Description

| Field | Description |
|---|---|
| `answer` | Final answer generated by the model |
| `citations` | References used by the answer |
| `tool_calls` | List of tool calls executed during the pipeline run |

The `tool_calls` field enables inspection of the execution trace and
supports reliability analysis of tool interactions.

### Example Output

``` json
{
  "answer": "Paris",
  "citations": ["wiki_france"],
  "tool_calls": []
}
```

------------------------------------------------------------------------

## Contract Validation

All outputs are validated using **JSON schema validation** before
metrics are computed.

Validation checks include:

-   required fields
-   correct data types
-   schema structure
-   presence of mandatory attributes

If validation fails, the run records a **schema compliance failure**.

### Contract Note

The JSON snippets shown in this document illustrate the conceptual
structure of the contracts.

Actual validation is performed using JSON schemas implemented inside the
pipeline validator.

------------------------------------------------------------------------

## Reliability Metrics

Contract validation directly affects several reliability metrics:

| Metric | Description |
|---|---|
| `schema_compliance_rate` | Percentage of outputs valid against schema |
| `tool_success_rate` | Successful tool executions |
| `policy_violation_rate` | Outputs violating system policies |

These metrics are stored in `metrics.json` for each run.

------------------------------------------------------------------------

## Failure Handling

When a contract is violated, the system records the failure and updates
the reliability metrics.

1. Records the event in `events.jsonl`
2. Marks the case as a reliability failure
3. Updates evaluation metrics

Typical contract violations include:

-   invalid JSON structure
-   missing required fields
-   malformed tool calls
-   incompatible output types

------------------------------------------------------------------------

## Design Principle: Contract-First Pipelines

The repository adopts a **contract-first architecture**.

Instead of treating LLM outputs as free text, the pipeline enforces
structured outputs that can be:

-   validated automatically
-   compared across runs
-   analyzed systematically

This design is essential for building **reliable, testable and observable
LLM systems**.
